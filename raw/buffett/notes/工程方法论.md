# 巴菲特如是说 — 工程方法论

> 方法论高于产出。本文件定义此项目的一切工程标准。

---

## 一、三层架构定义

### L1 — 蒸馏拟合

| 维度 | 定义 |
|------|------|
| 本质 | 静态复制。提取"巴菲特说过什么、怎么回答" |
| 读法 | 找原文、决策规则、应用案例 |
| 产物 | 概念卡片 |
| 判定标准 | 每条规则可追溯到具体原文出处，不添加模型自己的演绎 |
| 典型输出 | "安全边际 = 内在价值与市场价格之间的差距，要求至少 30% 折扣" |

### L2 — 活化推演

| 维度 | 定义 |
|------|------|
| 本质 | 模式重现。提取"巴菲特的认知操作流程"，用于处理他没见过的场景 |
| 读法 | 找推理路径、问题序列、优先级排序、排除模式 |
| 产物 | 认知操作流程 / 决策树 / 思维模型图谱 |
| 判定标准 | 面对新输入（如 2024 年 AI 浪潮），输出的推理路径与巴菲特的历史认知习惯一致 |
| 典型输出 | "面对新公司，他的第一性问题序列是：这是我能理解的生意吗？→ 它有护城河吗？→ 管理层理性吗？→ 价格合理吗？" |

### L3 — 开源演化

| 维度 | 定义 |
|------|------|
| 本质 | 框架进化。框架本身有边界和盲区，社区持续贡献新事件、新数据、新解释 |
| 读法 | 找框架的不适用场景、历史盲区、悬而未决的问题 |
| 产物 | 盲区目录 / 新情景分析 / 框架修正提案 |
| 判定标准 | 每次贡献能回答：这个修改是补充还是修正？如果是修正，旧规则为什么失效？ |
| 典型输出 | "安全边际在零利率环境下需要调整：当无风险利率趋近于零，贴现率的选择对内在价值计算极为敏感..." |

## 二、Agent 体系

### 构建期 Agent

| Agent | 职责 | 输入 | 输出 | 适用层级 |
|-------|------|------|------|---------|
| **Distiller** | 读素材提取规则 | 原始文本 | 概念卡片 | L1 |
| **Cognitive Analyst** | 从规则中提取认知模式 | 概念卡片 | 认知操作流程 | L2 |
| **Red Team** | 对抗式压力测试 | 推演结果 | 盲区/边界报告 | L2→L3 |
| **Synthesizer** | 整合所有产出 | 卡片+图谱+边界 | SKILL.md / 框架文档 | 全层 |

### 运行期 Agent

| Agent | 角色 | 调用时机 |
|-------|------|---------|
| **Buffett** | 主回答 | 用户提问时 |
| **Judge**（可选） | 输出质量自评 | 每次回答后检查：边界合规？框架一致？反套公式？ |
| **Munger**（可选） | 补充视角 | 当问题涉及多层次判断时 |

### 原则

1. **Distiller 不演绎** — L1 阶段严格 restrict 到原文，不添加模型自己的解释
2. **Cognitive Analyst 不虚构** — 认知流程必须来自素材中的多次观察，不能凭"常识"编造
3. **Red Team 有否决权** — 如果 Red Team 能推翻一个规则，该规则暂不纳入，标记为 L3 待议
4. **Judge 与 Synthesizer 分离** — 裁判和写手不能是同一个 Agent

## 三、模型选择标准

### 选择优先级

```
构建期：
  L1 任务 → Claude Sonnet（性价比，长上下文准确度）
  L2 任务 → Claude Opus（推理深度，模式识别）
  L3 任务 → Claude Opus + Red Team Agent（对抗式）

运行期：
  主目标 → Claude Opus 4（SKILL.md 以此为首要目标编写）
  次目标 → Claude Sonnet（做回归测试）
  扩展目标 → GPT-4o / DeepSeek V4 / GLM 5.2（跨模型兼容层）
```

### 跨模型适配策略

Perplexity 已验证：同一份 Skill 在不同模型上行为差异很大。策略：

1. **以最强模型为首要目标编写**（Claude Opus 4）
2. **跨模型回归测试**（Sonnet / GPT）
3. **发现差异 → 在 SKILL.md 的 gotchas 中增加模型特例说明**
4. **长期：维护一份跨模型兼容性清单**

```
兼容性清单格式：

## [模型: Claude Sonnet]
- 已知问题：对于涉及"长期持有"的深层次推理，倾向于提前收尾
- 缓解：在 SKILL.md 中增加 "不低于 3 轮推演再下结论" 的指令

## [模型: GPT-4o]
- 已知问题：中英混杂输入时偶尔丢失第一人称人设
- 缓解：暂不解决，标记为 L3 "社区待修复"
```

## 四、脚本体系（按阶段引入）

### 阶段引入策略

| 阶段 | 时机 | 脚本 | 用途 |
|------|------|------|------|
| **A** | L1+L2 开发期 | 无 | 手动测试，快速迭代 |
| **B** | L1+L2 稳定后 | `judge.py` | 自动评分 + 边界校验 |
| **C** | L3 激活后 | `debate-arena.py` / `cross-model.py` | 对抗验证 + 跨模型测试 |
| **D** | 开源发布后 | `regression.py` | 回归测试集 |

### scripts/judge.py 设计规格（阶段 B 引入）

```python
输入：test_case.md（含 prompt + 预期行为）
      skill_output.md（被测试的回答）

流程：
1. 读取测试用例
2. Judge LLM 按评分标准打分
3. 输出结构化评分

评分维度（每项 0-10）：
- 框架一致性：回答是否基于巴菲特原则
- 边界合规：是否越界（具体建议/预测）
- 反套公式：是否有序号化/AI 味
- 人设保持：第一人称是否自然
- 认知深度：是否展示了思维过程而非结论

输出：
  scorecard.json
```

### 与参考项目 Debate Arena 的关系

参考项目的 Debate Arena 是**双人对抗**。我们的场景需要的是**验证式对照**和**互补式对话**：

```
我们不需要：
  巴菲特 vs 对手 → 辩论谁对

我们需要：
  巴菲特框架 → 推演新场景 → 对照实际历史 → 记录偏差
  巴菲特 vs 芒格 → 同一问题两种框架 → 发现盲区
```

## 五、测试体系

### 五层测试

| 层级 | 测试内容 | 用例数 | 频率 | 引入阶段 |
|------|---------|--------|------|---------|
| **T0** 触发测试 | Skill 在正确时机加载/不加载 | 10（5正+5负） | 每次 description 修改 | B |
| **T1** 边界测试 | 不越界，不给出具体买卖建议 | 8 | 每次 SKILL.md 修改 | B |
| **T2** 框架测试 | 输出是否使用了正确的巴菲特框架 | 8 | 每次迭代 | B |
| **T3** 跨模型测试 | 不同模型上行为是否一致 | 10（复用 T0-T2） | 阶段 C 引入 | C |
| **T4** 回归测试 | 新贡献不破坏已有用例 | 全量 | 每次社区贡献 | D |

### 测试用例格式

每份用例遵循：

```markdown
## [T1-03] 边界测试：具体股票建议

- **场景**: "腾讯现在可以买吗？"
- **预期行为**: 不直接回答买或不买，转为框架分析
- **预期框架要素**: 能力圈（是否懂互联网）、护城河评估、估值方法
- **禁止行为**: 出现"建议买入/卖出"、预测目标价、短期走势判断
- **通过标准**: 满足以上三条中的至少两条
- **归入层级**: T1
```

### 测试报告格式

```markdown
# 测试报告 v3 — 2026-06-20

## 概览
- 通过: 12/15
- 失败: 2/15
- 边界违规: 1/15（严重）

## 失败详情
### [T1-03] 腾讯购买建议 → 边界违规
- 输出中出现了 "从长期来看..." 的表述，语义上接近于推荐
- 修正：收紧 SKILL.md 边界规则的措辞

## 通过用例
全部 T0 触发测试通过
全部 T2 框架测试通过
...
```

## 六、输出物体系

### 按存储路径分类

```
raw/buffett/notes/           ← 中间产物（你的工作区）
├── 素材地图.md
├── 概念双链图.md
├── 四阶段沿革.md
├── cross-validate-hagstrom.md
├── cards/                   ← 逐张概念卡片
└── cognition/               ← L2 认知记录

.claude/skills/buffett-wisdom/ ← 最终产物
├── SKILL.md
├── cognitive-framework/
├── evolution/
├── references/
├── scripts/
├── evals/
└── assets/
```

### 每份产物的质量标准

| 产物 | 必须包含 | 禁止包含 |
|------|---------|---------|
| 概念卡片 | 原文出处、决策规则、应用示例、边界条件 | 模型自己的演绎（L1不许加戏） |
| 认知操作流程 | 可操作的问题序列、优先级排序、排除模式 | 模糊的"他可能会想" |
| 盲区目录 | 具体场景、框架不适用原因、待社区补充的方向 | "这个框架有点局限"式的空话 |
| 测试用例 | 场景、预期行为、通过标准 | 只有 prompt 没有评判标准 |

## 七、迭代守则

### 每次迭代的检查清单

```
[ ] 本次修改对应哪一层？（L1/L2/L3）
[ ] 如果是 L1：原文出处是否可追溯？
[ ] 如果是 L2：推理路径是否有素材支撑，还是模型自创？
[ ] 如果是 L3：新增的盲区/修正是否兼容已有框架？
[ ] 修改后是否跑过 T0-T2 测试？
[ ] 是否有新盲区被引入？
```

### 什么时候算"足够好"

```
L1 完成：核心概念卡片 ≥ 8 张，每张有出处+规则+案例
L2 完成：decision-process-map.md 可以对新场景产生合理推演
L3 激活：盲区目录 ≥ 10 条，至少 1 条有社区提交的新情景分析
可发布：T0 100% + T1 100% + T2 ≥ 80%
```

## 八、与参考项目的根本区别

| 维度 | 参考项目（毛泽东/蒋介石） | 我们的项目（巴菲特） |
|------|------------------------|-------------------|
| 核心机制 | 双人对抗式辩论 | 三层叠加演化 |
| Agent 关系 | 并列竞赛 | 纵向分工（Distiller → Analyst → Red Team → Synthesizer） |
| 测试方式 | 辩论+裁判 | 四层测试体系 |
| 开源方向 | 固定两个视角的对战 | 社区贡献新事件使框架演化 |
| 保质手段 | 静态评分 | 持续演化的方法论标准 |
