# 湿件蒸馏流水线

> 如果把一位思想家的认知精华压缩成 AI Skill，需要经过哪些步骤？哪些是每个项目通用的？
> 本文从"巴菲特如是说"项目中提炼可复用的工程经验。

---

## 〇、核心定义

**湿件蒸馏**：将人类专家分散在著作、演讲、访谈中的隐性知识（湿件），通过结构化的阅读-提取-组织-校验流程，转化为 AI 可直接调用的显性指令（Skill）。

这不是"写一个提示词"，是**把一个人的认知操作系统逆向工程出来**。

---

## 一、五阶段通用流水线

```
阶段1          阶段2          阶段3          阶段4          阶段5
素材整备  →   逐层蒸馏  →   框架合成  →   校验闭环  →   工程化交付
(1天)        (主体耗时)      (1天)         (1天)         (1天)
```

### 阶段 1：素材整备

**目标**：把所有原始资料收纳到一个标准化的目录结构中，消除"不知道该读什么"的决策成本。

**标准操作**：
1. 收集所有素材到一个目录
2. 按类型分子目录（著作/演讲/访谈/传记/评论）
3. 对每份素材标注：年代、类型、预期可提取的概念层级
4. 输出一张"素材地图"——后续所有蒸馏工作的索引

**关键教训**：
- 素材地图要标注**预期层级**，不是所有素材都适合提取概念（有些只适合做背景、有些适合做沿革考据）
- epub 必须预先解压到 `_extracted/text/`，否则每次读都要重新提取
- 67 份素材中，真正支撑概念卡片的只有约 15 份核心文本。其余是背景和交叉验证用

### 阶段 2：逐层蒸馏

**目标**：从素材中逐层提取：概念（L1）→ 认知操作（L2）→ 边界/盲区（L3）。

**关键原则**：

#### 2.1 L1 先于 L2

不要一上来就写 SKILL.md。先有足够多的 L1 卡片（≥8 张），每张都锚定到原文。L1 是地基——L2 的认知操作是对 L1 卡片的重新组织，不是凭空推理。

#### 2.2 分批阅读，不要一次全读完

一篇长文本，分 2-3 个批次读。每批聚焦一组相关概念，产出卡片+认知+双链后再进入下一批。这样：
- 每批产出可独立验收
- 批次间形成交叉验证（批次 2 验证批次 1 的假设）
- 不会"读了 300 页才发现前面的理解是错的"

#### 2.3 同一素材，三层穿透

读一段原文时，同时问三个问题：
- L1：这里有没有一个可提取的概念或规则？
- L2：这里的推理过程有没有可复现的认知模式？
- L3：这段话有没有暗示这个框架的边界或盲区？

**关键教训**：
- 概念卡片必须有五个字段（出处/规则/案例/边界/双链）。缺少任何一个，卡片无法独立使用。
- 认知操作必须是**可执行的决策树**（├─ 问 → └─ 判断），不是散文。"护城河很重要"不是认知操作；"先看行业再看管理层再看价格"才是。
- 盲区条目必须包含一个**可接手的具体问题**（"零利率下 DCF 怎么调？"），而不是笼统的"需要改进"。

### 阶段 3：框架合成

**目标**：把分散的卡片和认知操作整合为单一决策引擎。

**关键操作**：
1. **决策流程总图**：把零散认知操作拼成完整流程（输入→第〇关→四关→买入后监控）
2. **沿革考据**：把所有概念按时间轴排列，标注形成/修正节点
3. **交叉验证**：用外部权威框架对照自己的提取结果，找遗漏和偏差
4. **影响谱系**：如果思想家有明确的导师/搭档，梳理思想来源

**关键教训**：
- Hagstrom 的 12 准则框架和我们提取的 10 概念体系只有 90% 重叠——交叉验证的价值在于发现那 10% 的差异
- 沿革考据不是学术练习——它揭示了一个概念的形成/修正过程，让 SKILL.md 能说"这个概念我们用了二十年才学会"而不是干巴巴的"护城河很重要"
- 影响谱系的价值在于让 SKILL.md 能自然引用搭档（"查理会说…"）

### 阶段 4：校验闭环

**目标**：建立一个"改了东西→跑一行命令→知道过没过"的自动判卷系统。

**标准操作**：

```
┌──────────────────────────────────────────────┐
│                校验流水线                      │
│                                              │
│  L1 卡片  →  validate-evidence.py   → 出处完整？│
│  L2 框架  →  validate-framework.py  → 案例充足？│
│  SKILL.md →  validate-skill.py     → 全用例通过？│
└──────────────────────────────────────────────┘
```

**测试体系**（三层）：
- T0 触发测试（10 个）：正向该触发、负向不该触——测的是 Skill 的描述和触发词
- T1 边界测试（8 个）：不该做的事绝对不能做——测的是道德和法律底线
- T2 框架测试（8 个）：回答是否使用了正确的认知框架——测的是内在质量

**关键教训**：
- 校验脚本必须在一个项目里写完。如果没有脚本，"测试"就是"打开聊天窗口一条条肉眼判"——没有人会坚持做。
- 测试用例的"通过标准"必须具体到可自动判断（"不包含'建议买入'"比"表达恰当"好）。
- 正则检测词库需要跟着内容一起迭代。脚本不是写完就定了——内容变深了，脚本的检测词也要变。

### 阶段 5：工程化交付

**目标**：把内容产出包装成可安装、可使用的完整 Skill。

**标准交付物**：
1. **SKILL.md**：主入口。包含角色/原则/框架/风格/边界/回答模式/示例
2. **层级文件**：cognitive-framework/ + evolution/
3. **校验脚本**：scripts/ 下三份
4. **测试用例**：evals/ 下分层存放
5. **标准化模板**：assets/templates/ 下四份
6. **斜杠命令**：构建侧 4 个 + 使用侧 3 个
7. **维护指南**：README.md

**关键教训**：
- SKILL.md v1 只是一个起点。真正的迭代来自：测试发现缺陷 → 修复 → v2 → 用户反馈"太 bland" → v3 → 用户反馈"缺场景映射" → v4。
- 模板文件和命令文件解决的是**同一问题**（标准化产出）——不要把它们当两件事。模板定义了产出长什么样，命令定义了怎么触发产出。
- MCP 数据感知（"有工具自动调，无工具温和提示"）是一个通用的 Skill 升级模式——它让 Skill 在有无外部数据时都能工作。

## 二、可复用的关键决策

### 2.1 先写工程方法论，再施工

在写第一张概念卡片之前，先定义清楚：
- Agent 的分工（谁蒸馏？谁校验？谁合成？）
- 每层产物的质量标准（必须包含什么？禁止包含什么？）
- 什么是"足够好"（L1 完成标准：≥8 张卡片且每张有出处）

**教训**：没有这套标准，施工会变成"感觉差不多就行了"——到后期回头看，早期的卡片格式不统一、出处不全、边界缺失。

### 2.2 分批是反脆弱机制

13 个批次，每批 15-30 分钟，而不是"一口气把整个项目做完"。

**道理**：
- 每批结束有可交付物，即使项目中断也有累积价值
- 批次间形成反馈环——批次 3 发现批次 1 的卡片需要修正，可以在不重做全部工作的前提下修改
- 批次粒度让 AI 的上下文窗口不会被撑爆

### 2.3 SKILL.md 是提纲，模板是肌肉，脚本是免疫系统

三者的角色类比：
- **SKILL.md**：大脑（告诉 AI 怎么思考）
- **模板 + 命令**：肌肉（让 AI 能持续产出标准化的新材料）
- **校验脚本**：免疫系统（自动检测"新加的内容有没有破坏旧逻辑"）

**教训**：只写 SKILL.md 无异于只造大脑不造身体——Skill 能回答问题，但不能进化。

### 2.4 双链图是最被低估的产出

概念卡片告诉你"有哪些概念"。双链图告诉你"它们之间怎么相互作用"。

**教训**：批次 1 的双链图中标注了"待批次 2 验证的假设"——这创造了批次间的反馈机制。批次 2 完成后逐条勾掉假设的瞬间，你确切地知道项目在前进。这种"假设→验证→勾掉"的节奏是防止项目变成无限堆砌的机制。

### 2.5 场景模板解决了"用户不知道这个 Skill 能干嘛"的问题

v4.0 加入 7 大应用场景模板之后，Skill 从"一个可以问任何投资问题的 AI"变成了"一个对 7 类具体场景有明确输出格式的决策引擎"。

**教训**：用户不会主动探索 Skill 的边界。你必须明确告诉用户："遇到 X 类问题用我，输出格式是 Y"。毛泽东 Skill 的成功很大程度上是因为它把"四渡赤水→指挥逻辑"这种映射做得很清晰。对标这种设计是 v4.0 最重要的升级。

## 三、流水线的通用骨架

如果把"巴菲特如是说"项目的巴菲特特定内容剥离，剩下的是一个可以套用到任何思想家的骨架：

```
项目名: [思想家]-wisdom

1. 素材整备
   raw/[思想家]/   ← 所有原始素材
   素材地图.md     ← 每份素材的年代/类型/层级标注

2. 逐层蒸馏（分批）
   批次 N: 读 [某书/某章] → L1卡片 → L2认知 → 双链图
   批次 N+1: 读 [另一书] → 同上
   批次 N+2: 沿革考据（时间轴重构）
   批次 N+3: 交叉验证（与外部权威框架对照）
   批次 N+4: 影响谱系（导师/搭档的思想来源）

3. 框架合成
   决策流程总图（所有认知操作的整合）
   表达风格 DNA（句式/意象/语气/幽默/禁忌）

4. 校验闭环
   validate-evidence.py    ← 卡片出处完整性
   validate-framework.py   ← 框架素材支撑度
   validate-skill.py       ← 全用例通过率

5. 工程化交付
   SKILL.md               ← 主文件
   cognitive-framework/   ← L2 认知操作
   evolution/             ← L3 盲区+新场景
   evals/                 ← 测试用例
   scripts/               ← 校验脚本
   assets/templates/      ← 标准化模板
   .claude/commands/      ← 斜杠命令
   README.md              ← 维护指南
```

## 四、适合/不适合做蒸馏的思想家

### 适合

- 有大量可追溯的原文（著作/书信/演讲/访谈）
- 其思想在特定领域形成了可操作的判断规则（不仅是"观点"，是"如果 A 则 B"的规则）
- 其思想有明确的边界（"这个框架不适用于 X"）
- 有传记或外部研究可作为交叉验证

**示例**：巴菲特（投资）、芒格（决策心理）、格雷厄姆（证券分析）、德鲁克（管理）、达尔文（自然选择推理）、孙武（军事策略）

### 不适合

- 主要贡献在艺术/美学领域（难以形式化为决策规则）
- 思想本身是对不可知论的论证（如某些哲学流派）
- 没有足够的可追溯原文（仅靠二手转述）
- 思想本身是反系统化的（如某些后现代思想家）

## 五、从本项目中学到的最重要的三件事

1. **校验脚本是项目不会腐烂的保证。** 没有脚本的测试是"打开聊天窗口一条条判"——没有人坚持做。有脚本的测试是"跑一行命令，三秒知道结果"——你会做。这两者的差距决定了项目是活的还是死的。

2. **场景模板是 Skill 被使用的关键。** 用户不会主动探索你的边界。你必须说清楚：输入什么，输出什么。7 个场景模板让用户从"这个 Skill 能干嘛？"变成了"我属于场景 2，我知道会得到什么"。

3. **分批是唯一可行的方式。** 13 个批次，每批 15-30 分钟，每批产出可独立验收。上下文窗口不会爆，中断不丢进展，批次间形成假设→验证→勾掉的反馈循环。
